# LabelHelp 项目文档

## 项目基础信息

- **项目名称**：LabelHelp（劳动与民事维权辅助系统）
- **项目定位**：面向普通用户的“咨询 + 检索 + 材料整理 + 文书生成/审查”一体化辅助工具
- **整体架构**：前后端分离开发，生产可一体化部署
  - **前端**：`Vite + React + React Router`（目录：`frontend/`）
  - **后端**：`FastAPI` 作为宿主，通过 `WSGI` 挂载 `Flask`（目录：`backend/`）
  - **数据存储**：`SQLite`（初始化与迁移：`backend/database/schema.py`）
- **关键依赖能力**
  - **大模型（LLM）**：用于咨询回答、检索改写、合同/文书审查、文书润色与字段抽取（配置：`backend/config/llm_config.json`）
  - **OCR**：用于图片/扫描件证据材料的文本识别（配置：`backend/config/tencent_ocr_config.json`）
- **本地运行方式（开发环境）**
  - **后端**：`python backend/fastapi_server.py`，默认 `http://localhost:8080`
  - **前端**：`frontend/` 下执行 `npm run dev`；构建 `npm run build` 产物到 `frontend/dist`，后端会优先托管该目录

## 项目需求分析

### 用户痛点

- **普通用户**：缺乏法律基础，不知道“该查什么法、该准备什么材料、证据如何整理、文书怎么写”
- **材料准备**：证据通常分散在聊天记录、图片、PDF、工资条、合同等不同载体中，整理成本高且易遗漏
- **检索体验**：用户表达口语化，直接检索容易“搜不到/搜不准”，需要自动改写与扩展
- **输出可用性**：生成内容若过长、格式混乱、引用不清，难以直接使用或进一步咨询律师

### 核心需求

- **咨询**：给出短而可读的回答，并提供可追溯的法条摘录引用
- **检索**：自动识别是查“案例/法条/混合”，支持改写扩展提升召回
- **案件化管理**：将用户问题沉淀为案件，围绕案件组织证据与处理进度
- **证据处理**：上传/识别/抽取/归档/导出，形成可交付材料包
- **合同与文书**：对合同/文书按清单结构化审查；草稿润色；模板化生成 DOCX
- **会话留痕**：咨询与检索过程可追溯（会话与消息落库）

## 解决方案思路

### 1) “咨询—检索—材料”三条链路并行但互相打通

- **咨询链路**：用户自然语言提问 → 自动拉取法规摘录 → LLM 生成短回答（含免责声明与引用）
- **检索链路**：识别检索意图（案例/法条/混合）→ 可选“改写 + 扩展”→ fanout 并行检索 → 去重合并与排序
- **材料链路**：案件 → 证据条目与附件 → OCR/文本抽取 → 证据类型推断 → 关系图谱构建 → 导出交付（CSV/ZIP）

### 2) 关键工程策略

- **检索增强**：使用“改写器 + 扩展 query + 并行 fanout + 去重合并”提升召回与准确度
- **可读输出约束**：咨询回答采用严格字数/格式约束，并进行后处理（去 emoji/标题/分割线等噪音）
- **低成本上下文承接**：会话每轮生成摘要，下一轮仅携带近若干轮摘要承接上下文
- **模板化文书生成**：DOCX 采用“模板 + 占位符映射”填充，保证格式稳定可下载提交
- **证据可追溯**：证据元数据修改、附件替换均记录历史版本（revisions），支撑回溯与协作

## 关键功能说明

### 1) 账号与鉴权

- **注册/登录/退出**：`/api/auth/register`、`/api/auth/login`、`/api/auth/logout`
- **个人资料维护**：`/api/auth/me`、`/api/auth/profile`、`/api/auth/change-password` 等
- **鉴权方式**：前端在请求头注入 `Authorization: Bearer <token>`（封装位置：`frontend/src/api.js`）

### 2) 法律咨询（短回答 + 引用）

- **入口**：`POST /api/chat`
- **返回**：`reply`（给用户看）、`citations/citation_refs`（引用与可点击索引）、`legal_index`（检索改写与关键词）
- **特点**：回答短、引用受控、含固定免责声明，并为会话压缩生成摘要块

### 3) 法规/案例检索（自动分流 + fanout）

- **入口**：`POST /api/retrieval`
- **能力**：
  - 自动识别检索类型（案例/法规/混合/其他）
  - 可选开启改写与扩展查询，触发 fanout 并行检索并合并去重
- **法规详情**：`GET /api/law-info?lawId=...`

### 4) 会话与消息（可追溯的咨询记录）

- **会话**：`/api/conversations`（创建/列表/删除/改标题）
- **消息**：`/api/conversations/<id>/messages`（发送/查询）
- **存储**：消息 `extra_data` 内存放 sender、引用、法律索引与摘要信息，便于 UI 展示与后续承接

### 5) 案件与证据库（上传/识别/分析/导出/图谱）

- **案件**
  - 创建/列表：`/api/cases`
  - 详情：`/api/cases/<case_id>`
- **证据（条目与附件）**
  - 列表/新增：`/api/cases/<case_id>/evidence`
  - 更新元数据：`/api/cases/<case_id>/evidence/<evidence_id>`
  - 上传附件并分析入库：`/api/cases/<case_id>/evidence/upload-analyze`
  - 图片 OCR（可选写库）：`/api/evidence/ocr`
  - 附件下载/上传：`/api/evidence/<evidence_id>/file`
  - 修订记录：`/api/cases/<case_id>/evidence/<evidence_id>/revisions`（含历史文件访问）
- **导出交付**
  - 证据清单：`/api/cases/<case_id>/evidence/export.csv`
  - 证据材料包：`/api/cases/<case_id>/evidence/export.zip`
- **证据关系图谱**
  - 查询：`/api/cases/<case_id>/evidence-graph`（返回图数据与 neo4j 结构）

### 6) 合同审查与摘要（清单覆盖式多轮编排）

- **清单审查**：`POST /api/contract-review/run`
  - 输出 `coverage`（每项 pass/fail/unclear/not_applicable）与 `risks`（风险点、原文片段、建议）
  - 多轮追问保证覆盖清单项或达到上限
- **结构化摘要**：`POST /api/contract-review/summary`

### 7) 文书辅助与生成（润色/抽取/模板 DOCX）

- **草稿润色**：`POST /api/documents/polish`
  - 约束：不编造新事实/新证据，不擅改金额与日期，只做规范化表达与结构优化
- **字段抽取（劳动监察投诉书）**：`POST /api/documents/extract-fields`
- **DOCX 生成**
  - 劳动监察投诉书：`POST /api/documents/labor-complaint-docx`
  - 民事起诉状：`POST /api/documents/civil-complaint-docx`
  - 申请执行书：`POST /api/documents/enforcement-application-docx`
  - 证据材料清单：`POST /api/documents/evidence-list-docx`
  - 劳动仲裁/调解申请书：对应 `/api/documents/labor-arbitration-application-docx`、`/api/documents/labor-mediation-application-docx`

### 8) 咨询资源聚合（12348 + FAQ）

- **12348 公开咨询代理**：`GET /api/public-consults`、`GET /api/public-consults/<pkid>`（带缓存）
- **FAQ**：`GET /api/consult-faqs`（分页、排序、关键词检索）

## 项目使用场景

### 场景 1：普通用户快速咨询并定位法律依据

- 用户输入问题（如欠薪、工伤、解除劳动合同）→ `POST /api/chat`
- 系统返回短回答 + 引用（可点开查看法规摘录）→ 用户据此明确下一步材料准备方向

### 场景 2：从“一个问题”沉淀为“一个案件”

- 创建案件（填写被申请人/案由/阶段）→ `POST /api/cases`
- 在案件下逐条录入/补充证据 → `POST /api/cases/<case_id>/evidence`

### 场景 3：证据材料上传与自动分析

- 上传图片/PDF/Word 等证据 → `POST /api/cases/<case_id>/evidence/upload-analyze`
- 系统完成 OCR/文本抽取与类型推断，形成可检索的证据条目，并可构建证据关系图

### 场景 4：材料导出交付给律师或线下提交

- 导出证据清单 CSV 与证据材料 ZIP → `export.csv` / `export.zip`
- 用于仲裁/诉讼材料准备、与律师协作、或归档留存

### 场景 5：合同/文书审查与生成

- 粘贴劳动合同正文 → `POST /api/contract-review/run` 获取风险点与清单覆盖
- 草稿润色或字段抽取后生成 DOCX → `POST /api/documents/*-docx` 下载可提交文书

## AI工具使用说明

### 使用到的 AI 能力与职责边界

- **法律咨询回答（LLM）**
  - 输入：用户问题 + 检索改写 + 法规摘录上下文
  - 输出：短回答（含固定免责声明）+ 引用标签 + 本轮摘要（供系统承接上下文）
  - 边界：不虚构条号与摘录之外的条文；内容仅供参考
- **检索 Query 改写（LLM）**
  - 产物：改写 query + 扩展 query 列表（用于 fanout 并行检索）
  - 目标：提升召回与命中率，避免用户口语化导致“搜不到”
- **合同/文书审查（LLM）**
  - 方式：按清单多轮追问直到覆盖（或达到轮次上限）
  - 输出：覆盖情况与风险建议，便于用户逐项修订或咨询律师
- **文书润色（LLM）**
  - 原则：仅规范化表达与结构，不编造事实/证据，不擅改金额与日期
- **字段抽取（LLM）**
  - 从自由文本抽取为结构化 JSON 字段，用于后续模板填充与 DOCX 生成

### OCR 的使用说明

- 场景：证据为图片或扫描件时，系统调用 OCR 提取文字，再进入“证据分析/类型推断/入库”链路
- 边界：OCR 结果可能存在误识别，关键金额、日期、姓名等仍需用户校对
