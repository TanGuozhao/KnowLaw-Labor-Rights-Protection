# 自动生成实体规则定义（草案）

本文档用于约束“自动知识图谱生成”中的**实体抽取**行为。后续代码实现应以本文件为准。

## 1. 范围与目标

- 输入：案件上下文（案由/阶段/诉求等）+ 材料标题 + 材料正文（OCR/描述）。
- 输出：结构化实体列表（`nodes`），供关系抽取与图谱入库使用。
- 目标：在劳动争议场景下，稳定提取关键主体、文书、事实、诉求与金额。

## 2. 实体类型（kind）定义

当前支持以下 `kind`，仅可从下列枚举中取值：

- `person`：自然人（劳动者、代理人、证人等）
- `organization`：组织机构（公司、单位、仲裁委、法院等）
- `document`：文书/证据材料（合同、工资条、聊天记录、通知书等）
- `fact`：客观事实（入职时间、离职时间、未签合同、未缴社保等）
- `claim`：主张/请求（赔偿请求、补缴请求、确认劳动关系等）
- `amount`：金额实体（具体金额、区间金额、工资标准等）
- `other`：无法归入以上类型但有图谱价值的实体

## 3. 实体字段定义

每个实体必须包含以下字段：

- `stable_key`：稳定键（英文小写 + 下划线，单次输出内唯一）
- `label`：实体展示名（中文，简洁可读）
- `kind`：实体类型（见上方枚举）

示例：

```json
{
  "stable_key": "employer_xxx_company",
  "label": "某某科技有限公司",
  "kind": "organization"
}
```

## 4. 抽取优先级规则

同一段材料中，实体抽取优先级如下（从高到低）：

1. 劳动关系核心主体（劳动者、用人单位）
2. 直接证据文书（合同、工资流水、考勤、社保记录、通知）
3. 关键争议事实（入离职、工资拖欠、加班、违法解除）
4. 明确诉求（补偿金、双倍工资、经济赔偿等）
5. 可量化金额/标准（具体金额、月工资基数）

说明：

- 正文较短时，至少保证主体 + 文书（或标题指向实体）被抽取。
- 信息噪声较多时，优先保留能支撑劳动争议论证链条的实体。

## 5. 命名与归一化规则

### 5.1 stable_key 生成

- 只允许：`a-z`、`0-9`、`_`
- 连续分隔符折叠为单个 `_`
- 去除首尾 `_`
- 最大长度建议：64
- 若冲突，追加后缀 `_1`、`_2`...

### 5.2 label 规范

- 保留中文原语义，避免过度缩写
- 去除首尾空白
- 避免使用“未命名”“实体”等无信息名称（除非无可用信息）

### 5.3 类型纠偏

- 明确是公司/单位/委员会/法院 -> `organization`
- 明确是合同/通知/记录/截图 -> `document`
- 明确是金额数值 -> `amount`
- “请求/主张/诉请/要求”语义优先归为 `claim`
- 时间点或事件描述优先归为 `fact`

## 6. 去重与合并规则（抽取阶段）

- 单次输出中，语义等价实体应合并，不重复创建。
- 若同名但类型冲突，按以下优先顺序决策：
  - `person`/`organization` > `document` > `fact`/`claim` > `other`
- 若实体仅指代模糊代词（如“其”“该方”）且无法消解，可不抽取。

## 7. 场景化最低覆盖要求

在劳动争议场景下，每份材料尽量满足：

- 至少 1 个主体实体（`person` 或 `organization`）
- 至少 1 个文书或事实实体（`document` 或 `fact`）
- 若出现金额表达，至少 1 个 `amount`
- 若出现明确诉求语句，至少 1 个 `claim`

## 8. 输出格式约束（给模型）

- 只输出一个 JSON 对象，不要 Markdown，不要解释文本。
- 顶层结构：

```json
{
  "nodes": [
    { "stable_key": "string", "label": "string", "kind": "person|organization|document|fact|claim|amount|other" }
  ],
  "edges": []
}
```

说明：本文件先聚焦“实体规则”，`edges` 暂可为空数组或由关系规则文档另行约束。

## 9. 质量检查清单

提交前逐项检查：

- `stable_key` 是否唯一
- `kind` 是否在允许枚举内
- `label` 是否可读且非空
- 是否覆盖主体/文书/事实等关键实体
- 是否存在明显重复或类型误判

## 10. 待你修改的建议位置

你可以重点改这几节来驱动后续代码实现：

- 第 2 节：实体类型枚举是否增删
- 第 4 节：抽取优先级是否调整
- 第 5/6 节：归一化与去重策略
- 第 7 节：最低覆盖阈值
- 第 8 节：最终输出 JSON 约束
